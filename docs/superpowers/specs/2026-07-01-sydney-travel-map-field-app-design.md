# 시드니 건축답사 지도 현장 배포 앱 설계

## 목적

이 프로젝트는 기존 시드니 건축답사 지도 앱을 건축학과 학생과 교수가 현장에서 바로 사용할 수 있는 완성형 정적 웹앱으로 다듬는다. 학생들은 휴대폰으로 날짜별 코스와 장소 설명을 확인하고, 교수는 같은 앱으로 이동 동선과 건축 관찰 포인트를 설명할 수 있어야 한다.

이번 단계에서는 현재의 가벼운 구조를 유지한다.

- `index.html`
- `style.css`
- `app.js`

React, Vue 같은 프레임워크로 갈아타지 않는다. 빌드 시스템을 추가하기보다 신뢰성, 가독성, 현장 사용성을 우선한다.

## 사용자

주 사용자는 시드니 건축답사를 따라가는 건축학과 학생이다. 교수는 같은 화면을 수업 안내와 현장 설명 자료로 사용한다.

따라서 화면은 화려한 홍보 페이지보다 실용적인 필드 가이드에 가까워야 한다. 빠르게 읽히고, 야외 모바일 화면에서도 조작하기 쉬우며, 장소의 건축적 의미가 명확해야 한다.

## 범위

### 포함

- 화면에 보이는 한글 깨짐을 수정하고 UI 문구를 자연스러운 한국어로 정리한다.
- Leaflet 기반 지도와 정적 웹 배포 구조를 유지한다.
- 날짜/테마 코스 선택, 일정 카드, 지도 보기, 주요 버튼의 모바일 사용성을 개선한다.
- Day 1부터 Day 5까지의 날짜별 코스와 테마별 코스를 유지한다.
- UI에 드러나는 장소 설명을 학생용 답사 자료처럼 읽히도록 정리한다.
- 기존 지리적 근접도 기반 동선 최적화를 유지하되 예외 상황 안내를 개선한다.
- 장소 추가 흐름에서 검색 중, 성공, 실패 상태를 더 명확히 보여준다.
- 이미지 저장 결과에 제목, 코스 설명, 범례, 파일명이 잘 반영되도록 개선한다.
- GitHub Pages식 정적 배포를 위한 간단한 README를 추가한다.

### 제외

- React, Vue 등 프레임워크 기반 재구축
- 서버 저장소, 사용자 계정, 실시간 공동 편집
- 지도 타일까지 포함하는 완전한 오프라인 모드
- 유료 라우팅 API 또는 API 키 기반 서비스
- 대중교통 시간표까지 반영하는 복잡한 시간표 최적화

## 제품 동작

### 코스 탐색

사용자는 날짜별 코스와 테마별 코스를 전환할 수 있다. 코스를 선택하면 다음 요소가 함께 갱신된다.

- 지도 마커
- 동선 라인
- 일정 목록
- 코스 제목과 설명
- 이미지 저장용 제목과 부제

`전체 일정` 화면은 개요와 탐색용이다. 동선 최적화는 특정 날짜 또는 특정 테마 코스를 선택했을 때만 실행한다.

### 지도와 일정 목록

지도는 공간 관계를 보는 핵심 화면이고, 일정 목록은 읽기 중심의 답사 자료다. 사용자가 카드만 훑어봐도 다음 정보를 이해할 수 있어야 한다.

- 장소명
- 방문 순서 또는 시간
- 장소 유형
- 짧은 건축 관찰 포인트
- 식당이나 서비스 장소의 선택적 연락처

마커는 기존처럼 건축, 식당/카페, 기타 장소를 구분한다.

### 동선 최적화

기존 근접 이웃 방식의 지리적 정렬을 유지한다. 클라이언트만으로 작동하는 가벼운 현장 플래너에는 이 방식이 적합하다.

최적화는 다음 원칙을 따른다.

- 첫 번째 장소는 출발점으로 유지한다.
- 나머지 장소를 현재 위치에서 가장 가까운 다음 목적지 순으로 재정렬한다.
- 지도와 일정 목록을 함께 갱신한다.
- 완료 후 사용자가 이해할 수 있는 한국어 확인 메시지를 보여준다.

다음 예외 상황은 친절하게 안내한다.

- `전체 일정` 화면에서 최적화를 실행한 경우
- 장소가 3개 미만인 경우
- 좌표가 없는 장소가 포함된 경우

### 장소 추가

사용자는 건축/장소, 식당/카페, 기타 장소를 추가할 수 있다. 폼은 현재 선택된 장소 유형과 필요한 입력값을 명확히 보여줘야 한다.

Nominatim으로 위치를 검색할 때는 다음 상태를 처리한다.

- 검색 중임을 표시한다.
- 좌표를 찾은 뒤에만 장소를 추가한다.
- 결과가 없거나 요청이 실패하면 이해하기 쉬운 오류를 보여준다.
- 장소 추가 후 현재 코스 화면과 지도 상태를 일관되게 유지한다.

### 이미지 저장

이미지 저장 버튼은 학생들과 공유하기 좋은 지도 이미지를 만든다. 저장 이미지에는 다음이 포함되어야 한다.

- 코스 제목
- 코스 부제 또는 설명
- 마커와 동선이 보이는 지도
- 범례

파일명에는 코스 키 또는 읽기 쉬운 코스명이 들어간다.

## UI 설계

### 데스크톱

데스크톱은 기존의 2분할 구조를 유지한다.

- 왼쪽 사이드바: 앱 제목, 코스 탭, 장소 추가 폼, 일정 목록, 이미지 저장 버튼
- 오른쪽 지도 영역: 지도, 코스 오버레이 제목, 범례

불필요한 장식보다 빠른 스캔과 반복 사용을 우선한다.

### 모바일

모바일은 현장 사용성을 우선한다.

- 가로 스크롤이 생기지 않는다.
- 탭과 버튼은 손가락으로 누르기 편한 크기다.
- 일정 카드는 야외에서도 읽기 쉽다.
- 지도는 충분한 세로 공간을 가진다.
- 긴 한국어 버튼과 탭 문구는 넘치지 않고 자연스럽게 줄바꿈된다.

정확한 반응형 배치는 구현 중 기존 CSS 구조에 맞춰 정하되, 휴대폰 폭 화면에서 반드시 검증한다.

## 데이터와 구조

코스 데이터는 계속 `app.js`의 정적 JavaScript 객체로 둔다. 현재 프로젝트 규모에서는 이 방식이 충분하고, GitHub Pages 배포도 단순하게 유지된다.

`app.js` 안에서는 다음 책임이 명확히 나뉘도록 유지한다.

- 코스 데이터
- 지도 초기화
- 코스 렌더링
- 동선 최적화
- 장소 추가 폼 처리
- 지오코딩
- 이미지 저장

구현 중 특정 함수가 지나치게 복잡해지면 작은 헬퍼 함수로 나눌 수 있다. 다만 이번 단계에서 대규모 리팩터링은 하지 않는다.

## 오류 처리

앱은 다음 상황에서 자연스러운 한국어 피드백을 제공한다.

- 최적화할 수 없는 상태
- 위치 검색 실패
- 필수 입력 누락
- 이미지 저장 실패

일반 사용 흐름에서 깨진 한글, 원시 기술 오류, 애매한 placeholder 문구가 보여서는 안 된다.

## 검증

이번 정적 앱 개선은 수동 검증으로 충분하다. 완료 전에 다음을 확인한다.

- `index.html`이 로컬에서 열린다.
- Day 1부터 Day 5까지 오류 없이 렌더링된다.
- 테마 코스가 렌더링된다.
- `전체 일정` 화면이 렌더링된다.
- 장소 추가 폼의 유형 탭이 정상 전환된다.
- 특정 코스에서 동선 최적화가 작동한다.
- `전체 일정`에서 최적화 실행 시 명확한 안내가 나온다.
- 모바일 폭 화면에서 레이아웃이 사용 가능하다.
- 이미지 저장 버튼이 다운로드 흐름을 실행한다.

가능하면 인앱 브라우저 또는 로컬 정적 서버로 시각 검증한다.

## 배포

앱은 정적 GitHub Pages 사이트로 배포 가능한 상태를 유지한다. README에는 다음을 간단히 설명한다.

- 앱의 목적
- 로컬에서 여는 방법
- GitHub Pages 배포 방법
- Leaflet, OpenStreetMap 타일, Font Awesome, Google Fonts, html2canvas, Nominatim 같은 외부 의존성

## 결정 사항

막힌 제품 결정은 없다. 사용자가 승인한 보수적인 정적 앱 안정화 방향으로 구현 계획을 작성한다.
